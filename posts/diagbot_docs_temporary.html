DiagBot

A production-ready Agentic AI Orchestrator for Bangladeshi diagnostic centers — built on LangGraph, ChromaDB, and SQLite. Handles multi-turn conversations for test inquiries, preparation guidance, appointment booking, and report status tracking with a hard-wired safety loop that cannot be bypassed.

Python 3.11+ LangGraph 0.2+ LangChain 0.3+ ChromaDB 0.5+ Claude Sonnet 4 MIT License
What makes DiagBot different?

The preparation safety loop is enforced at the graph edge level, not in a prompt. No appointment for a fasting test can be written to the database until the patient has explicitly acknowledged the preparation instructions — regardless of what the user sends.

Installation

Prerequisites

  • Python 3.11 or higher — type hints and async features require it
  • pip 23+ — for dependency resolution
  • Google API key — get one at makersuite.google.com
  • ~500 MB disk space for ChromaDB ONNX embedding model (downloaded on first run)
  • Internet connection for first run only (model caching)

  1. Clone the repository

    Get the source code onto your machine.

bash

  1. Create and activate a virtual environment

  1. Configure your API key

    Copy the environment template and fill in your Google key.

# Optional overrides:

# LOG_LEVEL=DEBUG

  1. Run DiagBot

    Databases are created automatically on first launch.

System architecture

DiagBot is composed of four Python modules and two persistence back-ends, wired together by a LangGraph StateGraph. The graph is the single orchestration layer — all routing decisions happen inside it.

📐

schema.py

Central TypedDict state definitions. Everything the graph knows about the current session lives here.

🗄️

database.py

SQLite appointment store + ChromaDB vector knowledge base. Auto-initialises on first run.

🔧

tools.py

Three async @tool functions: RAG search, booking engine, and report status tracker.

🕸️

main.py

LangGraph StateGraph with 8 nodes, 6 routing functions, and the interactive CLI runner.

Graph topology

The graph has eight nodes connected by conditional edges. Every patient message enters at intake_node and exits at END — the path between them depends entirely on the state flags accumulated during the conversation.

Safety invariant: The path from intake_node to booking_node via the direct "booking" edge is only available when selected_tests contains no fasting tests OR prep_confirmed = True. The instruction_node path is taken otherwise. This is a conditional edge function — not a prompt instruction.

Data flow

A single patient message triggers the following sequence of operations inside the graph:

1

Message appended

The user's text becomes a HumanMessage and is appended to state.messages via the add_messages reducer. The graph never replaces the message list — it only grows.

↓
2

intake_node runs

Keyword heuristics scan the message for intent signals. If ambiguous, a lightweight LLM call returns a single-token classification. An invoice ID regex extracts any DX-XXXXXX pattern into state.invoice_id.

↓
3

Conditional router dispatches

route_from_intake() reads state.intent, state.selected_tests, and state.prep_confirmed to decide the next node. This is a pure Python function with no LLM involvement.

↓
4

Specialist node invokes LLM

The LLM receives the full conversation history plus a specialist system prompt. It decides autonomously whether to call a tool or respond directly. If tools are called, an AIMessage with tool_calls is returned.

↓
5

ToolNode executes

LangGraph's built-in ToolNode runs each tool call asynchronously. Results are appended as ToolMessage objects. The next routing function sees both the tool call and its result.

↓
6

sync_state_node reconciles

Scans ToolMessage contents to extract structured data — confirmed invoice IDs, etc. — and merges them back into the typed state fields. No LLM is involved here.

↓
7

State returned to caller

The updated DiagBotState is returned to the session loop. The CLI runner prints the last AIMessage and feeds the same state object back into the next invocation.

schema.py — State definitions

This module defines the single source of truth that flows through every node in the graph. All fields are explicitly typed via TypedDict.

DiagBotState

The main state object. LangGraph passes this between nodes — each node receives the full state and returns a partial update dict. The graph merges updates using field-level reducers.

messages List[BaseMessage] Full conversation history. Uses the add_messages reducer — nodes append new messages, the graph merges automatically. Never replaced wholesale.
patient_info PatientInfo Progressively collected name and phone. Optional keys — may be partially filled.
selected_tests List[str] Catalog IDs of tests the patient is interested in (e.g. ["lipid_001", "cbc_001"]).
prep_confirmed bool Safety-loop gate. Booking is blocked for fasting tests until this is True. Only instruction_node can set it.
intent Optional[str] Classified routing intent. One of: "inquiry", "booking", "status", "general".
pending_instructions Optional[str] Preparation text awaiting patient acknowledgement. Set by instruction_node, cleared after confirmation.
invoice_id Optional[str] Extracted or generated invoice reference (e.g. "DX-F3K9P2"). Used for status lookups.
error_message Optional[str] Human-readable error from last failed validation. Cleared on success.

Manages two entirely separate storage back-ends. Both initialise automatically — no migrations, no manual setup needed.

ChromaDB — Test knowledge base

A persistent local vector store at data/chroma_store/. Uses ChromaDB's built-in ONNX embedding model — no external API key required. Each test is stored as a rich bilingual document string enabling semantic search in both English and Bangla.

tools.py — LangChain tools

Three async @tool decorated functions that form the action layer. All are bound to the LLM via .bind_tools() — the agent decides when and how to call them.

search_diagnostic_tests

signature
async def search_diagnostic_tests(query: str, top_k: int = 3) -> str
  • Performs cosine-similarity search against the ChromaDB collection
  • Accepts queries in English or Bangla — the ONNX embedding model handles both
  • Filters results with relevance score < 0.25 to avoid low-quality matches
  • Returns formatted markdown with price (BDT), preparation, turnaround, and fasting flag

book_appointment

signature
async def book_appointment(
    patient_name: str,
    phone: str,
    test_ids: List[str],
    appointment_date: Optional[str] = None
) -> str
  • Validates patient name (≥ 2 chars), phone (11-digit BD format: 01[3-9]\d{8}), and catalog IDs
  • Generates a unique DX-XXXXXX invoice ID using random.choices
  • Defaults appointment_date to the next working day (respects BD Friday skip)
  • Writes atomically to SQLite inside a context manager — partial writes cannot occur
  • Returns a full confirmation card including total cost in BDT
Guard: This tool should only be called when prep_confirmed = True for fasting tests. The graph's conditional edge enforces this, but the tool also has a defensive double-check inside booking_node.

check_report_status

signature
async def check_report_status(invoice_id: str) -> str
  • Normalises the invoice ID to uppercase before querying (case-insensitive)
  • Returns status badge, patient name, appointment date, and test list
  • Provides status-specific messaging: collection instructions for Ready, helpline for Delayed
  • Returns a clear not-found message if the invoice ID doesn't exist

Builds the LangGraph StateGraph, registers all nodes, defines conditional edges, and exposes a CLI runner. The graph is compiled once with graph.compile() and is then stateless between invocations — all session state lives in the DiagBotState dict passed by the caller.

Conditional edge functions

main.py — primary router
def route_from_intake(state: DiagBotState) -> str:
    intent = state.get("intent", INTENT_GENERAL)
    if intent == INTENT_INQUIRY:
        return "rag_node"
    if intent == INTENT_STATUS:
        return "status_node"
    if intent == INTENT_BOOKING:
        catalog = {t["id"]: t for t in TEST_CATALOG}
        selected = state.get("selected_tests", [])
        needs_prep = any(
            catalog.get(tid, {}).get("requires_fasting", False)
            for tid in selected
        )
        if needs_prep and not state.get("prep_confirmed", False):
            return "instruction_node"  # safety gate
        return "booking_node"
    return "general_node"

The safety loop

The preparation safety loop is DiagBot's most critical clinical feature. It enforces a hard architectural guarantee that cannot be bypassed by prompt injection or user trickery.

Why this is enforced at the edge level

A naive implementation might add a check inside booking_node or write a system prompt instruction like "never book until prep is confirmed". Both approaches can be circumvented — the former by routing bugs, the latter by prompt injection.

DiagBot instead enforces the invariant in route_from_intake(), which is a pure Python function that reads the typed state. No LLM is involved in this decision. The LangGraph runtime calls this function and cannot proceed to booking_node unless the function returns "booking_node". This makes the safety guarantee as strong as the Python interpreter itself.

Agentic reasoning

DiagBot's intelligence emerges from six interlocking agentic patterns — not from a single large prompt.

1. Two-tier intent classification

The intake_node applies fast keyword heuristics first (covering ~90% of Bangladeshi diagnostic queries in English and Bangla). Only when the heuristic returns no clear signal is a lightweight LLM call made — with a strict max_tokens=10 constraint that forces a single-word response. This keeps median latency low while handling edge cases gracefully.

2. Retrieval-augmented generation

The LLM never hard-codes test information. Every price, preparation instruction, and turnaround time is retrieved from ChromaDB at query time. The ONNX embedding model handles bilingual queries — "sugar test", "ব্লাড সুগার", and "fasting glucose" all resolve to the correct test entries via cosine similarity.

3. The preparation safety invariant

The prep_confirmed boolean is a state machine invariant enforced by a routing function. See the safety loop section for details.

4. Progressive state accumulation

LangGraph's add_messages reducer means the agent accumulates context across turns without any "memory" abstraction. Patient name, phone, test selections, and invoice IDs persist in DiagBotState and are available to every subsequent node. The agent never asks for information it already has.

5. Autonomous tool invocation

All three tools are bound to the LLM via .bind_tools(). The model decides autonomously when to call a tool, what arguments to extract from the conversation, and how to synthesise the result into a natural reply. For example, the agent will extract a catalog ID from a previous search_diagnostic_tests result and pass it directly to book_appointment — without being instructed to do so.

6. Graceful degradation

All tool implementations catch exceptions internally and return formatted error strings rather than raising. The LLM receives these as tool results and composes a recovery message. The graph never crashes — it always produces an output to the patient.

RAG pipeline

The Retrieval-Augmented Generation pipeline runs entirely locally with no external embedding API.

RAG pipeline sequence
Patient query → ChromaDB ONNX encoder → query vector
             → cosine similarity against 12 document vectors
             → top-k results filtered by relevance > 0.25
             → formatted markdown string returned to LLM
             → LLM synthesises natural-language response

Embedding strategy

Each test entry is embedded as a rich 200–300 character document that includes the test name in English and Bangla, category, description, preparation instructions, and price. This multi-field document approach means a query for "চর্বি পরীক্ষা" (fat test) matches "Lipid Profile / লিপিড প্রোফাইল" because the embedding captures semantic overlap across both languages simultaneously.

Programmatic usage

You can invoke the graph directly from your own code without using the CLI runner:

python
import asyncio
from langchain_core.messages import HumanMessage
from main import build_graph, create_initial_state
from database import init_databases

async def single_turn(user_text: str) -> str:
    init_databases()
    app   = build_graph()
    state = create_initial_state()
    state["messages"].append(HumanMessage(content=user_text))

    result = await app.ainvoke(state)

    ai_msgs = [m for m in result["messages"] if hasattr(m, "content")
               and m.__class__.__name__ == "AIMessage"]
    return ai_msgs[-1].content if ai_msgs else ""

response = asyncio.run(single_turn("CBC test er price koto?"))
print(response)

Error handling

DiagBot follows a no-crash guarantee — every error surface is caught and returned as a formatted string that the LLM can synthesise into a patient-friendly message.

ScenarioHandlerPatient-visible message
Invalid phone number book_appointment tool ❌ Please provide an 11-digit Bangladeshi mobile number starting with 01
Unknown catalog ID book_appointment tool ❌ Unknown test ID(s): xyz. Use search to find valid IDs.
Invoice not found check_report_status tool ❌ Invoice ID not found. Please double-check the ID on your slip.
SQLite write failure book_appointment tool ❌ Booking failed (database error): [message]
ChromaDB unavailable search_diagnostic_tests tool ❌ Search error: [message]
Graph invocation error CLI runner try/except ❌ An unexpected error occurred: [message]
Low-relevance search results search_diagnostic_tests — relevance filter ℹ️ No closely related tests found. Please try rephrasing.
